모델을 어떻게 정의하나요?

Express는 데이터베이스와 관련된 기능들을 전혀 갖고 있지 않으며, 이는 third-party 모듈에 의존합니다. 따라서 거의 모든 데이터베이스의 인터페이스 사용을 가능하게 합니다.

어떻게 인증을 처리하나요?

이 또한 Express가 스스로 지원하는 부분은 아니므로 사용 하고 싶은 인증 솔루션을 사용하시면 됩니다. 간단한 username / password 스킴을 보기 위해서는 example를 참고하시기 바랍니다.

Express가 지원하는 템플릿 엔진(template engines)은 무엇인가요?

(path, locals, callback) signature 형태를 가진 것이라면 어떤 것이든 지원 가능합니다. 템플릿 엔진(template engines)과 캐싱을 normalize 하기 위해서, consolidate.js 프로젝트를 확인 해 볼 것을 권합니다. 리스트에 포함 되지 않은 템플릿 엔진도 Express signature를 지원 할 가능성이 높습니다.

제 애플리케이션을 어떻게 구조화(structure) 시켜야 하나요?

이 질문에 대한 명확한 답은 없습니다. 이는 당신의 애플리케이션의 규모와 관련된 팀에 달려 있기 때문입니다. 최대한의 유연함을 위해 Express는 구조(structure)에 관해서는 예외사항을 두지 않습니다.

라우트들과 구체적 애플리케이션 논리는 원하는 디렉토리 구조 안에 원하는 만큼 많이 포함 시킬 수 있습니다. 아래 예제를 참고 해 보세요.

  • Route listings
  • Route map
  • Route bootstrapping
  • MVC style controllers

이러한 패턴들을 단순화 시키는 Express의 third-party 익스텐션 또한 사용 가능합니다.

  • Resourceful routing
  • Namespaced routing

여러 디렉토리에서 statics를 어떻게 사용(serve)하나요?

일반적으로 애플리케이션 안에서 미들웨어를 여러 번 사용합니다. 다음 미들웨어 setup과 "GET /javascripts/jquery.js"으로의 요청은 처음으로는 "./public/javascripts/jquery.js"을 검사하고, 만약 이것이 존재하지 않는다면 그 이후의 미들웨어는 "./files/javascripts/jquery.js"을 검사합니다. 

app.use(express.static('public'));
app.use(express.static('files'));

statics을 사용(serving)하기 위해서는 경로 이름을 어떻게 prefix 해야 하나요?

Connect의 일반적인 "mounting"은 prefix 문자열이 경로에 포함되지 않은 것 처럼 효과적인 기능을 하며 경로 이름에 어떤 미들웨어가 호출 될 것인지의 "prefix"를 정의 할 수 있도록 도와줍니다., 예를 들어, "GET /files/javascripts/jquery.js"가 필요 할 때, req.url는 미들웨어가 파일을 사용 할 수 있도록 "/javascripts/jquery.js"을 노출 시킴으로서 "/files"에서 미들웨어를 마운트 할 수 있습니다. 

app.use('/public', express.static('public'));

Express 2.x 애플리케이션을 어떻게 옮길 수(migrate) 있나요?

Express 2x는 거의 대부분 node 1.0으로 지원 되기 때문에, Express 3x가 소개하는 대로 refactoring과 API를 바꾸면서 까지 업데이트를 할 필요는 없습니다. 그러므로 2x에 만족을 한다면, 2x에 남아 계셔도 큰 불편은 없습니다. 더 많은 정보나 기타 사항은 migration의 위키 페이지나 3.x로 만들어진 list of changes를 참고하세요.

404를 어떻게 처리 해야 하나요?

Express에서 404는 에러의 결과가 아닙니다. 그러므로 오류 핸들러(error handler) 미들웨어는 404를 캡쳐하지 않으며, 이는 404가 단순히 더 필요한 작업이 부족하기 때문에 나타나는 것이기 때문입니다. 다시 말하면 Express는 이 때에 모든 미들웨어와 라우트를 이미 수행 한 상태이며, 아무런 응답을 받지 못한 상태입니다. 그래서 404를 처리 하고자 한다면, 다른 미들웨어들의 가장 아래에 또 다른 미들웨어 하나를 더 추가하시면 됩니다. 

app.use(function(req, res, next){
  res.send(404, 'Sorry cant find that!');
});

Express에서 오류 핸들러(Error Handler)를 어떻게 설정 해야 하나요?

오류 처리 미들웨어는 다른 보통 미들웨어와 마찬가지로 정의되지만, 특히 오류 처리 미들웨어는 4개의 argument (err, req, res, next)로 정의되어야 합니다. 

app.use(function(err, req, res, next){
  console.error(err.stack);
  res.send(500, 'Something broke!');
});

더 많은 정보는 error-handling를 참고하세요.

단순(plain) HTML을 어떻게 만들 수(render) 있나요?

할 필요가 없습니다! res.render()을 HTML와 "render" 할 필요가 없습니다. 특정한 파일을 사용 해야 한다면, res.sendfile()를 사용하고, 디렉토리의 많은 assets를 이용(serving)한다면, express.static() 미들웨어를 사용하세요.

Express 코드 베이스(codebase)는 얼마나 큰 가요?

Express는 3.0.0 릴리즈가 932 SLOC인 매우 작은 프레임 워크입니다. Express가 built 되는 Connect의 필수 비율은 276 SLOC밖에 되지 않습니다. Connect와의 선택적 번들 미들웨어는 추가적으로 1143 SLOC을 더해주며 이는 느린 로딩(lazy loaded)으로 사용됩니다.
